# AI開発ルール

## プロジェクト概要

- Flutter / Dart / Firebaseで開発するバレーボールチーム運営アプリ。
- 対象はWeb、Android、Windows。
- 認証にはFirebase Authentication、データにはCloud Firestore、Web公開にはFirebase Hostingを使用する。

## 基本方針

- 作業前にREADME、関連ドキュメント、関連コードを確認する。
- 実装状態を確認せず、推測だけでコードを変更しない。
- 情報不足で結果が大きく変わる場合は、実装前に質問する。
- 影響の小さい点は、仮定した内容を明記して進める。
- 既存機能、既存デザイン、データ構造、認証、権限、レスポンシブ対応を維持する。
- 依頼されていないUI変更、機能追加、依存関係追加、リファクタリングを行わない。
- 関係のないファイルを変更しない。
- ファイル削除や破壊的なGit操作を勝手に実行しない。

## 作業開始時

- `git status`を確認する。
- 未コミット変更はユーザーの作業として扱い、勝手に破棄しない。
- 関連ファイルと既存実装を確認する。
- 複数ファイルに影響する変更では、先に実装計画を提示する。

## 実装

- 変更範囲を必要最小限にする。
- 既存の命名規則、Riverpodの状態管理、設計、コードスタイルを維持する。
- 新しい依存関係は、必要性を説明して許可を得てから追加する。
- Firebase Authentication、Firestoreのデータ構造・権限・セキュリティルールの変更は特に慎重に扱う。
- Firebaseの設定値や秘密情報をログ、回答、コミットへ含めない。
- FirestoreやStorageのルールを推測で作成・変更しない。

## 確認

通常は次を実行する。

```powershell
flutter pub get
flutter analyze
flutter test
flutter build web
```

- `git diff`で意図しない変更がないことを確認する。
- 実行できなかった確認項目は、実行したように扱わず明記する。

## Git・デプロイ

- ユーザーの明示的な指示なしにcommit、push、merge、deployを行わない。
- mainへ直接変更を入れず、作業ブランチを使用する。
- `git reset --hard`、`git clean`、force pushなどの破壊的操作を行わない。
- Firebaseへのデプロイは、対象プロジェクトと環境を確認し、ユーザーの許可を得てから行う。

## 完了報告

以下を必ず報告する。

1. 変更したファイル
2. 実装した内容
3. 確認・テストした内容
4. 残っている問題
5. ユーザーが次に実行するコマンド
